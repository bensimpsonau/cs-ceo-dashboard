Comment & Feedback Guidelines

  • Be specific: "Change the hook to lead with the number" not "make it better"
  • Use Asana comments: All feedback lives on the task. Not Slack, not text messages.
  • Tag the person: @mention the person who needs to action the feedback
  • One round max: If it needs more than one round of revisions, jump on a 5-minute call
  • Time-box: Reviews must be completed within 4 hours of assignment (business hours)

Version Control

  • Every revision gets uploaded as a new attachment on the Asana task (v1, v2, v3)
  • The approved version is pinned as a comment with "FINAL" label
  • Never overwrite previous versions. Keep the trail.
  • Google Drive links in the task always point to the latest version

Ben's Final Approval Checklist

Every piece of content must pass all five checks:

  • Voice Check: Does this sound like me / CS? Would I say this in a conversation?
  • Brand Check: No "crypto" (use "digital assets"). No em dashes. No hype language. No urgency tactics.
  • CTA Check: Correct ManyChat keyword? Link goes to collectiveshift.co/cryptoblueprint?
  • Accuracy Check: Any claims backed by data? 116% CAGR noted as "verified and audited by Crypto Tax Australia"?
  • Platform Check: Right format, dimensions, and character count for the target platform?

Jarvis in the Review Process

Before any content reaches a human reviewer, Jarvis runs a brand compliance scan.
  • Checks for banned words: "crypto" (should be "digital assets"), em dashes, "to the moon", "WAGMI", "not financial advice" (CS never uses this disclaimer), hype language
  • Checks CTA: ManyChat keyword matches the content pillar
  • Checks voice: Compares against Ben's Voice Profile for tone and style markers
  • Flags issues as comments on the Asana task before it hits Ben's queue
  • This saves Ben 2-3 minutes per piece (across 61 pieces/week = ~3 hours saved)

Emergency Publishing Protocol

For market crashes, regulatory news, major hacks, or breaking events:

  1. Ben triggers. No one else decides to go to emergency mode.
  2. Jarvis drafts a take within 15 minutes.
  3. Ben edits + approves in a single pass (5-10 minutes).
  4. Post order: X/Twitter (fastest) then Instagram Story then Reel (if warranted).
  5. Asana: Jarvis creates task, moves directly to Published section, tags as P1 Urgent.
  6. Optional: Same-day EDM if Ben decides the event warrants it.
  7. Rule: Speed does not override quality. Voice and brand checks still apply.

05 Content Calendar Integration

Mapping the 90-Day Calendar into Asana

The Evergreen Content System contains 103+ content concepts organized by pillar and format. Here is how to bring that into Asana:

  1. Jarvis batch-creates all 103 concepts as tasks in the Content Calendar project via the Asana API. Each task has Content Pillar, Content Type, Platform, and Person fields pre-filled.
  2. Assign to weeks: Spread across 13 weeks (90 days). Jarvis sets the Publish Date field for each task.
  3. Calendar View: Switch to Calendar View to see the full 90 days laid out visually. Color by Content Pillar for balance at a glance.
  4. Each Monday standup: Review that week's tasks in Asana. Move from "Idea" to "Draft" to kick off production.
  5. Recycle: At week 14, the cycle restarts. Jarvis refreshes evergreen content with updated data and re-creates tasks.
Jarvis can populate the entire 90-day calendar in one batch. One command. 103+ tasks created with all custom fields, descriptions, subtasks, and publish dates. The team opens Asana Monday morning and the next 13 weeks are planned.

Conflict Resolution

When two pieces compete for the same time slot:

  1. Time-sensitive wins. Market events, product launches, and limited-time content always take priority.
  2. Higher-pillar wins. Protect/Strategy content takes priority over Community/Results content.
  3. The bumped piece moves to the next available slot on the same platform, same day if possible.
  4. Never stack. Do not post two pieces on the same platform within 2 hours of each other (algorithm penalty).
  5. Ben breaks ties. If it is not obvious, Ben decides. This should happen less than once per week.

06 Team Onboarding & SOPs

Editor SOP (Reel + YouTube Editing)

Reel Editing (Georgio / Future Hire)

  1. Check Asana: filter "In Edit" section tasks assigned to you each morning
  2. Download raw footage from Google Drive link in the task
  3. Edit per CS standards: vertical 9:16, auto-captions (white text, dark background), light music if appropriate
  4. Keep reels under 60 seconds. Ideal: 30-45 seconds.
  5. First 2 seconds must match the hook line in the task brief
  6. Export at 1080x1920, H.264, high quality
  7. Upload final to Google Drive + attach to Asana task
  8. Move task to "In Review" section
  9. Turnaround target: 4 hours per reel, all 21 Ben reels done by Wednesday EOD

YouTube Long-Form Editing (Future Hire)

  1. Download raw footage from Google Drive
  2. Edit for pacing: cut dead air, add B-roll where relevant, add lower-thirds with presenter name/title
  3. Add CS intro bumper (3 seconds) and outro card with CTA
  4. Color grade to match CS brand look (clean, slightly warm)
  5. Export at 1920x1080, H.264
  6. Mark 3 best clip timestamps in task comments (for reel extraction)
  7. Upload to Google Drive + attach to Asana task
  8. Turnaround: 48 hours from filming session

Copy Reviewer SOP

  1. Open the Asana task. Read the copy in the description or attached document.
  2. Run through the Brand Compliance Checklist (below)
  3. Check voice: Does this sound like Ben? Or the brand it is meant to represent?
  4. Check facts: Are stats current? Is the 116% CAGR noted as "verified and audited by Crypto Tax Australia"?
  5. Leave specific feedback as Asana comments. Tag the original author.
  6. If the copy is good, move task to "In Review" section (for Ben's final approval)
  7. Time target: 15 minutes per review

Ben's SOP (Final Approver)

  1. Morning block (9:30am, 20 min): Open "Ben's Approval Queue" view in Asana. Review and approve/return everything from overnight.
  2. Afternoon block (2:00pm, 15 min): Second pass for anything that came in during the day.
  3. Approval: Move task to "Approved" section. Add a thumbs-up reaction to the asset comment.
  4. Rejection: Add a comment with specific feedback. Move task back to "In Edit" or "Draft" section.
  5. Batch reels: After Sunday filming + editing, approve all 21 reels in one 30-minute session (Wed or Thu).
  6. Goal: Nothing sits in "In Review" for more than 8 business hours.

Content Creator SOP (Travis, Sarn, Matt)

  1. Check Asana every morning. Your "By Person" view shows everything assigned to you.
  2. When creating content: move the Asana task to the correct section at every stage (Draft, In Edit, etc.)
  3. All raw footage goes to Google Drive. Link it in the task.
  4. Write your own caption first. Jarvis can polish it, but your expertise must be the foundation.
  5. Move to "In Review" section when you believe it is ready. Do not send half-finished work for review.

🤖 Jarvis + Asana API Management

Jarvis connects directly to the Asana API to manage the entire content board. No manual data entry. No copy-pasting. The AI runs the production engine while the team focuses on creating.

What Jarvis Can Do via Asana API

Create Tasks

Batch-create 100+ tasks from the 90-day content calendar. Each task includes title, description, custom fields (type, pillar, platform, person), subtasks, and publish date. One command populates a month of content.

Update Statuses

Move tasks between sections (Idea, Draft, In Edit, In Review, Approved, Scheduled, Published) and update the Status custom field to match. Keep the board current without manual dragging.

Add Draft Copy

Post AI-generated first drafts as Asana comments on tasks. Tweets, LinkedIn posts, EDM copy, reel scripts, and captions. The creator opens the task and finds a ready-to-edit draft waiting.

Weekly Reports

Query all tasks published in the past 7 days. Count by platform, pillar, and person. Compare to the 61/week target. Generate a summary and post to Discord or as an Asana project status update.

Auto-Assign

When creating tasks, Jarvis assigns them based on content type rules. Reels go to Ben or Travis. YouTube goes to Matt/Travis/Sarn. EDMs go to Jarvis (draft) then Ben (approve). Carousels go to Collaborative.

Brand Compliance

Scan task descriptions and comments for banned words ("crypto", em dashes, hype language). Flag issues as comments before Ben reviews. Acts as an automated pre-review layer.

Setup: Connect Jarvis to Asana

Step 1: Generate a Personal Access Token (PAT)

  1. Go to https://app.asana.com/0/developer-console
  2. Click "Create new token"
  3. Name it "Jarvis Bot" (or "ClawBot")
  4. Copy the token. Store it securely (1Password or environment variable).
  5. This token gives Jarvis read/write access to all projects in the workspace.

Step 2: Find Your Project GID

  1. Open the Content Calendar project in Asana
  2. Look at the URL: https://app.asana.com/0/PROJECT_GID/...
  3. The number after /0/ is the project GID. Copy it.
  4. Do the same for Email Production and YouTube Pipeline projects.

Step 3: Find Section GIDs

  1. Use the API to list sections: GET /projects/{project_gid}/sections
  2. Note the GID for each section (Idea, Draft, In Edit, In Review, Approved, Scheduled, Published)
  3. Jarvis uses these GIDs to move tasks between sections.

Step 4: Find Custom Field GIDs

  1. Use the API to list custom fields: GET /projects/{project_gid}/custom_field_settings
  2. Note the GID for each custom field and its enum option GIDs (for dropdown values).
  3. Jarvis uses these to set Content Type, Pillar, Platform, Person, etc. on every task.

Zapier / Make Alternative

If you prefer not to use the Asana API directly, you can achieve most of the same functionality with Zapier or Make.com:

  • Zapier: "Create Task in Asana" action. Supports custom fields, sections, assignees. Good for simple automations (e.g., form submission creates task).
  • Make.com: More powerful Asana module. Supports batch operations, custom field updates, section moves, and comment creation. Better for complex workflows.
  • Limitation: External tools add latency and cost. The direct API approach via Jarvis is faster, free, and more flexible.

Recommendation: Use the Asana API via Jarvis for everything. Use Make.com only for the Customer.io integration (where you need webhook triggers that Jarvis does not run 24/7).

Security & Access

  • PAT Scope: The Personal Access Token has access to everything in the workspace. Use a dedicated Asana account for Jarvis if you want to limit scope.
  • Token Storage: Store the PAT in an environment variable (ASANA_PAT) or in 1Password. Never commit it to code.

  • Audit Trail: All Jarvis actions show up in Asana as created/modified by the PAT owner. Full visibility for the team.
